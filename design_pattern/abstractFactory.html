# 抽象工厂模式

# 定义：抽象工厂模式的实质是“提供接口，创建一系列相关或独立的对象，而不指定这些对象的具体类。”

​ 抽象工厂与工厂方法最大的区别：抽象工厂中每个工厂可以创建一组产品，而工厂方法只能创建一个。这一组产品称为产品族，产品族中的每一个产品部分属于一个产品等级结构。

# 产品组与产品等级结构

​ 产品等级结构可以理解为 产品的继承结构。 比如说抽象类是鞋子，子类包括Nike的鞋子Adidas的鞋子，Puma鞋子，这时候鞋子与具体的鞋子之间就构成了产品等级结构。

​ 产品族。在抽象工厂中产品族是指由 同一工厂生产的产品，隶属于不同的产品等级结构中。 比如： 苹果公司，有苹果手机，苹果电脑，苹果平板电脑等。这些产品就构成了一个产品族。

# 抽象工厂类图

角色 关系 作用
AbstractProduct（抽象产品） 具体产品的父类 抽数具体产品的接口
Product（具体产品） 工厂类创建的目标类 描述生产的具体产品
AbstractFactory（抽象工厂） 具体工厂的父类 描述具体工厂的公共接口
ConcreteFactory（具体工厂） 被外界调用的类 描述具体的工厂，实现创建产品的对象实例

# 例子

package AbstractFactory;

// class CPU
abstract class CPU {
    public abstract void show();
}

// class EmberCPU
class EmberCPU extends CPU {
    public void show() {
        System.out.println("生产出了EmberCPU");
    }
}

// class EnginolaCPU
class EnginolaCPU extends CPU {
    public void show() {
        System.out.println("生产出了EnginolaCPU");
    }
}

// class MMU
abstract class MMU {
    public abstract void show();
}

// class EmberMMU
class EmberMMU extends MMU {
    public void show() {
        System.out.println("生产出了EmberMMU");
    }
}

// class EnginolaMMU
class EnginolaMMU extends MMU {
    public void show() {
        System.out.println("生产出了EnginolaMMU");
    }
}

// class EmberFactory
class EmberFactory extends AbstractFactory {
    @Override
    public CPU createCPU() {
        return new EmberCPU();
    }

    @Override
    public MMU createMMU() {
        return new EmberMMU();
    }
}

// class EnginolaFactory
class EnginolaFactory extends AbstractFactory {
    @Override
    public CPU createCPU() {
        return new EnginolaCPU();
    }

    @Override
    public MMU createMMU() {
        return new EnginolaMMU();
    }
}

enum Architecture {
    ENGINOLA, EMBER
}

abstract class AbstractFactory {
    public abstract CPU createCPU();
    public abstract MMU createMMU();
}

public class Client {
    public static void main(String[] args) {
        EmberFactory emberFactory = new EmberFactory();
        CPU cpu = emberFactory.createCPU();
        cpu.show();
    }
}

# 优点：

  1. 降低耦合

    抽象工厂模式将具体的产品创建延迟到具体工厂子类中，这样将对象封装了起来，可以减少客户端和具体产品类之间的依赖。从而使系统耦合度降低，这样更有利于后期的维护和扩展。

  2. 更符合开闭原则

    新增一个产品类时，只需要增加相应地具体产品类和相应地工厂子类

  3. 更符合单一职责原则

    每个具体的工厂类只负责创建对应的产品。

# 缺点

​ 抽象工厂模式很难支持新种类产品的变化。 这是因为抽象工厂接口中已经确定了可以被创建的产品集合，如果需要添加新产品，此时就必须去修改抽象工厂的接口，这样就涉及到抽象工厂类的以及所有子类的改变，这样也就违背了“开发——封闭”原则。

# 应用场景

  • 这个系统有多个系列产品，而系统中只消费其中某一系列产品
  • 系统要求提供一个产品类的库，所有产品以同样的接口出现，客户端不需要依赖具体实现。
