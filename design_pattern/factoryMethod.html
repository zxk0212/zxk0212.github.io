# 工厂方法模式

工厂方法是一种实现了“工厂”概念的面向对象设计模式。

# 定义：定义一个创建对象的接口，但让实现这个接口的类来决定实例化哪个类。工厂方法让类的实例化推迟到子类中进行。

# 工厂方法类图

# 模式组成

角色 关系 作用
Product(抽象产品) 具体产品的父类 描述具体产品的公共接口
Concrete Product(具体产品) 抽象产品的子类，工厂类创建的目标 描述生产的具体产品
Creator(抽象工厂) 具体工厂的父类 描述具体工厂的公共接口
Concrete Creator(具体工厂) 抽象工厂的子类，被外界调用 描述具体工厂，实现创建产品实例的方法

# 使用步骤

步骤1： 创建抽象工厂类，定义具体工厂的公共接口； 步骤2： 创建抽象产品类 ，定义具体产品的公共接口； 步骤3： 创建具体产品类（继承抽象产品类） & 定义生产的具体产品； 步骤4：创建具体工厂类（继承抽象工厂类），定义创建对应具体产品实例的方法； 步骤5：外界通过调用具体工厂类的方法，从而创建不同具体产品类的实例

# 例子

大众汽车公司，它旗下也有不少汽车品牌。大众汽车公司就好比一个汽车工厂，负责生产和销售汽车。它可以为客户提供一个客户需要的汽车。但是，大众汽车也会不定时推出新款汽车，在这种情况下，简单工厂模式需要，在添加一种汽车的情况下，同时要修改公司内部的生产环境（也就是工厂类的代码）。简单而言，就是工厂类（汽车公司）什么都要干，要修改必须大动干戈。因而一定程度上违背了开闭原则。而工厂方法模式则不一样，大众汽车公司不在总公司生产汽车，而是成立分公司，收购别的公司，成立具有针对性的汽车工厂专门生产对应的汽车。若客户的大量需求得不到满足，则总公司就另外成立新的二级公司(新品牌汽车的工厂)生产汽车，从而在不修改具体工厂的情况下引进新的产品。

package Factory;

public class FactoryMethod {
    public static void main(String[] args) {
        FactoryVW f1 = new FactoryVW();
        Car car = f1.Manufacture();
        car.Show();
    }
}
// 步骤1： 创建抽象工厂类，定义具体工厂的公共接口

interface Factory {
    public abstract Car Manufacture();
}

// 步骤2： 创建抽象产品类 ，定义具体产品的公共接口
interface Car {
    public abstract void Show();
}

// 步骤3： 创建具体产品类（继承抽象产品类）， 定义生产的具体产品
    //具体产品大众汽车类
class VW implements Car {
    @Override
    public void Show() {
        System.out.println("生产出了大众汽车");
    }
}

    //具体产品奥迪类
class Audi implements Car {

    @Override
    public void Show() {
        System.out.println("生产出了奥迪汽车");
    }
}

    //具体产品保时捷
class Porsche implements Car {

    @Override
    public void Show() {
        System.out.println("生产出了保时捷汽车");
    }
}

    //具体产品兰博基尼
class Lamborghini implements Car {

    @Override
    public void Show() {
        System.out.println("生产出了兰博基尼汽车");
    }
}

//步骤4：创建具体工厂类（继承抽象工厂类），定义创建对应具体产品实例的方法；
class FactoryVW implements Factory{
    @Override
    public Car Manufacture() {
        return new VW();
    }
}

class FactoryAudi implements Factory{
    @Override
    public Car Manufacture() {
        return new Audi();
    }
}

class FactoryPorsche implements Factory{
    @Override
    public Car Manufacture() {
        return new Porsche();
    }
}

class FactoryLamborghini implements Factory{
    @Override
    public Car Manufacture() {
        return new Lamborghini();
    }
}

# 相较于简单工厂的优点

  1. 更符合开闭原则

    新增一种产品时只需要增加相应地具体产品类和相应的工厂子类即可

  2. 符合单一职责原则

    每个具体的工厂类只负责创建对应的产品

  3. 不使用静态工厂方法，可以形成基于继承实现的等级结构

# 使用场景

  • 当一个类不知道它所需要的对象的类时 在工厂方法模式中，客户端不需要知道具体产品类的类名，只需要知道所对应的工厂即可；
  • 当一个类希望通过其子类来指定创建对象时 在工厂方法模式中，对于抽象工厂类只需要提供一个创建产品的接口，而由其子类来确定具体要创建的对象，利用面向对象的多态性和里氏代换原则，在程序运行时，子类对象将覆盖父类对象，从而使得系统更容易扩展。
  • 将创建对象的任务委托给多个工厂子类中的某一个，客户端在使用时可以无须关心是哪一个工厂子类创建产品子类，需要时再动态指定，可将具体工厂类的类名存储在配置文件或数据库中。
